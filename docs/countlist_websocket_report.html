Inventory Count Lists

WebSocket Architecture & Flow Report

Overview of how count-list updates propagate between browser modules and Django Channels consumers, plus opportunities to streamline or centralize the current implementation.

High-Level Flow

  • Editing a count row updates totals locally, assembles the payload in sendCountRecordChange, then posts it across the list-specific socket (app/core/static/core/js/objects/pageObjects.js:1117 + app/core/static/core/js/objects/webSocketObjects.js:186).
  • CountListConsumer persists the change and rebroadcasts the payload to all clients watching the list (app/core/consumers.py:49-133).
  • CountListWebSocket listeners merge the update back into the DOM, refreshing container details if the modal is visible (app/core/static/core/js/objects/webSocketObjects.js:273-420).
  • Add/delete operations use the same loop: client sends add_count / delete_count, consumer mutates DB and CountCollectionLink, and broadcasts count_added/count_deleted.
  • Count list metadata edits travel through CountCollectionWebSocket and the CountCollectionConsumer broadcast path, keeping the links page and detail page in sync.

    function updateDate(eventTarget){

                msg.recordId === recordId &&

                return; // Skip processing this message entirely

        // Record this message as processed

            recordId: recordId,
            action_type: data.data.action_type,
            containerId: data.data.containerId,
            timestamp: Date.now()
        });
        
        // Limit the size of the received messages map

            this.receivedMessages.delete(oldestKey);

        try {

            let countedQuantity = data['data']['counted_quantity'];

    constructor(countListWebSocket) {
        this.webSocket = countListWebSocket;
        this.cachedContainers = new Map(); // Store container data by countRecordId

    /**
     * Gets container data for a count record, either from cache or from the server
     * @param {string|number} countRecordId - The ID of the count record
     * @param {string} recordType - The type of record (blend, blendcomponent, etc.)
     * @param {boolean} forceRefresh - Whether to force a refresh from the server
     * @returns {Array} - Array of container objects
     */
    getContainers(countRecordId, recordType, forceRefresh = false) {
        // Check cache first if not forcing refresh
        if (!forceRefresh && this.cachedContainers.has(countRecordId)) {
            // CM Log
            return this.cachedContainers.get(countRecordId);
        }

        // Get from server
        let containers = [];
                $.ajax({
            url: `/core/get-json-containers-from-count?countRecordId=${countRecordId}&recordType=${recordType}`,
            async: false, // Critical: This must be synchronous to ensure we get the data before returning
                    dataType: 'json',
                    success: function(data) {
                containers = data;
            },
            error: function(xhr, status, error) {
                console.error(`❌ Error fetching containers from server: ${error}`);
            }
        });

        // Update cache with the latest data from server
        this.cachedContainers.set(countRecordId, containers);

        return containers;
    }

Backend Channels & Services

  • CountListConsumer centralizes list actions, trusting client payloads for fields like expected quantity, variance, and container arrays; also updates ItemLocation when present (app/core/consumers.py:49-185).
  • add_count_to_db returns the dict expected by the frontend, but currently assumes ItemLocation always exists—missing rows raise exceptions.
  • delete_count_from_db trims CountCollectionLink.count_id_list with no guard against stale IDs or duplicates.
  • CountCollectionConsumer manages rename/delete/reorder broadcasts; automated list generation lives in inventory_services.create_automated_countlist and sends collection_added updates system-wide.

Pain Points & Standardization Opportunities

  • Heavy reliance on globals (window.thisCountListWebSocket, window.countListPage, window.containerManager) makes sequencing brittle and hinders reuse.
  • ContainerManager fetches synchronously even though websocket payloads already contain containers; consolidating on the socket data would avoid redundant requests and UI stalls.
  • Duplicated event bindings between setUpEventListeners and _setupSingleRowEventHandlers risk double submissions when rows are inserted dynamically.
  • Message schemas differ: some broadcasts wrap data under data, others expose fields at the top level, forcing defensive code.
  • Row creation hinges on cloning existing DOM nodes with imperative tweaks; makes UI changes tedious and error-prone.
  • Backend assumes companion records (e.g., ItemLocation) exist; a missing row can bubble up into a socket exception.

Template Strategy Thoughts

  • Server-rendered partials would centralize row markup, but require sending HTML over the socket and make each update a render + diff step.
  • A client-side templating approach (e.g., stored <template> or precompiled snippets) offers DRY structure while keeping the websocket JSON payload.
  • Consider templating the container modal first—currently the most repetitive block— to gauge the benefit before migrating full rows.

Suggested Next Steps

  1. Define a formal JSON contract for websocket messages (including containers) and have both CountListWebSocket and CountListConsumer enforce it, enabling removal of synchronous container fetches.
  2. Extract a reusable client template for count rows or container rows and refactor addCountRecordToUI to use it, paving the way for optional backend templating later.
  3. Introduce a small controller module that owns the shared websocket and page state, letting other modules import it instead of touching globals directly.
